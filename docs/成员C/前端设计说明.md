# 成员 C 前端设计说明（对齐更新后作业要求）

**依据文档**：`docs/作业要求updated.md`  
**角色**：C — Frontend & UX  
**目标**：说明前端应如何设计，才能同时满足作业强制要求、加分项，以及评分维度中的 Usability / 交互式审查能力。

---

## 一、作业要求对前端的核心约束

### 1.1 功能要求分级

| 等级 | FR | 作业要求 | 前端必须做什么 |
|------|-----|----------|----------------|
| **强制** | FR 1.0 | 多源需求输入（CSV/文本/直接输入） | 上传区 + 可选文本输入框 |
| **强制** | FR 1.1 | 解析 Input Fields / Data Ranges / Conditions / Expected Actions | 结构化结果表 + 字段缺失提示 |
| **强制** | FR 2.0 | 风险评分与测试优先级（High/Medium/Low） | 风险矩阵/热力图 + 每条需求的优先级展示 |
| **强制** | FR 3.0 | 至少 3 种黑盒技术（EP/BVA/DT）自动生成用例 | 用例表 + 按 technique 区分 + standard_ref |
| **强制** | FR 6.0 | JSON/Excel/CSV 导出测试产物 | 导出按钮 + 字段完整性提示 |
| **强制** | **交互式审查** | 设计者可审查、修订、修改覆盖项/策略/用例 | **当前最大缺口，必须补强** |
| **加分** | FR 4.0 | 状态机建模 + 覆盖准则路径 | FSM 可视化 + 覆盖路径列表 |
| **加分** | FR 5.0 | Oracle / Expected Result 合成 | 置信度 + 人工复核 UI |
| **加分** | FR 7.0 | 基于风险或覆盖效率的套件优化 | Before/After + 优化模式切换 |

> **结论**：你现有的 4 步 Pipeline 骨架方向正确，但要从「展示型 Demo」升级为「可交互的设计工作台」，才能满足作业中 **Mainly** 段落的强制要求。

### 1.2 非功能要求（NFR）对前端的影响

| NFR | 作业说明 | 前端设计对策 |
|-----|----------|--------------|
| **Usability** | 明确列入 NFR | 侧边 Stepper 流程清晰；每步有标题、说明、下一步引导；错误/加载态完整 |
| **Performance** | 用例生成建议 ≤ 2s（尽力达成） | 按钮 loading、骨架屏、超时提示；慢请求时展示「预计耗时」而非白屏 |
| **Maintainability** | 技术可维护 | 保持 `types/` + `services/api.ts` 分层；页面按 Step 拆组件 |
| **Security** | 后端负责为主 | 前端不硬编码 AUT 地址；敏感配置走环境变量 |

### 1.3 评分维度与前端的关系

作业评分中 **Coverage & Effectiveness (40%)** 权重最高。对 C 而言，这意味着：

- 演示视频里必须能完整走通：**输入需求 → 看风险 → 生成用例 → 人工改一条 → 导出**
- 不能只有 AI 自动生成结果，必须体现 **human tester（designer）参与并验证有效性**
- 前端要支持 **traceability（可追溯）**：用例 ↔ 需求 ↔ 覆盖项 ↔ 测试技术，能在 UI 上看见关联

## 二、推荐的前端信息架构（在现有 4 步基础上演进）

### 2.1 总体布局（保持 Pipeline，增强交互层）

```
┌──────────────────┬────────────────────────────────────────────────────┐
│  左侧 Stepper     │  顶栏：当前项目 / 目标应用名称 / Live·Mock 状态      │
│                  ├────────────────────────────────────────────────────┤
│  1 Input & Parse │  【主工作区】                                        │
│  2 Risk Analysis  │   - 上半：AI 生成结果（只读或半只读）                  │
│  3 Generate &     │   - 下半：Designer Review Panel（可编辑）★ 新增重点   │
│    Evaluate      │   - 右侧抽屉：Traceability / History（可选）          │
│  4 Optimize &     │                                                    │
│    Export        │  【底栏】上一步 / 下一步 / 保存修订 / 导出              │
└──────────────────┴────────────────────────────────────────────────────┘
```

**设计原则**：

1. **流程不变**：继续用 4 步 Stepper，与演示脚本一致，评委易理解。
2. **每步都留「人工修订区」**：满足 Mainly 段落的 interactive review。
3. **加分 FR 不单独占主导航**：FR4/5/7 嵌入 Step 3/4，避免菜单过多。

### 2.2 与作业「Mainly」流程的映射

作业强调的设计流程：

```
Concept
  → Coverage Item Identification
  → Coverage Strategy & Method
  → Test Cases + Traceability
  → Prompt Design
  → Results Analysis
  → Improvement with Evidence
```

建议在前端用 **「设计工作台」子面板** 承载，而不是再增加 6 个顶级菜单：

| 作业流程阶段 | 建议落点（前端 Step） | UI 形态 |
|--------------|----------------------|---------|
| Concept / 需求理解 | Step 1 | 原始需求 + 解析字段表 |
| Coverage Item Identification | Step 2 或 Step 3 上半区 | 可编辑列表：覆盖项 ID、描述、关联 requirement_id |
| Coverage Strategy & Method | Step 3 上半区 | 每条需求选择 EP/BVA/DT/FSM；显示推荐理由（来自 RAG） |
| Test Cases + Traceability | Step 3 主表格 | 用例表 + 列：requirement_id / technique / coverage_item_id |
| Prompt Design | Step 3 侧栏或 Modal | 只读展示当前 Prompt 摘要（B 提供）；演示用 |
| Results Analysis | Step 3 Oracle 区 + Step 4 优化对比 | 置信度、Pass/Fail、缩减率 |
| Improvement with Evidence | Step 3/4 底部「修订记录」 | 显示：修改前/修改后、新增覆盖项、新增用例条数 |

## 三、分 Step 详细设计说明

### Step 1 — Input & Parse（FR 1.0 / FR 1.1）【强制】

**目标**：让 designer 导入目标应用（AUT）需求，并完成结构化。

| 区域 | 组件 | 必做 | 建议增强（满足交互审查） |
|------|------|------|--------------------------|
| 输入区 | `Upload.Dragger` + 文本粘贴框 | ✅ 已有上传 | 增加「直接粘贴需求文本」入口（作业明确提到 direct input） |
| 解析触发 | 主按钮 + loading | ✅ 已有 | 解析完成后自动展开结果区 |
| 结果表 | Table：requirement_id, input_fields, data_ranges, conditions, expected_action, confidence, missing_fields | ✅ 已有 | **行内编辑**：允许 designer 修正 LLM 解析错误的字段 |
| 校验 | Alert 展示 missing_fields | ✅ 已有 | 编辑后重新校验 / 标记「人工已确认」 |
| 状态 | Live / Mock 标签 | ✅ 已有 | 保持 |

**接口**：`POST /ingest` + `POST /parse`（A 暴露，B 保证 parse 质量）  
**演示要点**：上传 `aut_15_requirements` 或粘贴 AUT SRS 片段 → 展示结构化字段 → **手动改一条 expected_action** → 进入 Step 2。

---

### Step 2 — Risk Analysis（FR 2.0）【强制】

**目标**：为每条需求赋予 Risk Score 与 Test Priority。

| 区域 | 组件 | 必做 | 建议增强 |
|------|------|------|----------|
| 热力图 | 5×5 Impact × Likelihood 矩阵 | ✅ 已有 | 点击单元格筛选对应需求 |
| 快照 | 高/中/低风险数量 | ✅ 已有 | 与热力图联动 |
| 明细表 | requirement_id, score, level | ✅ 已有 | **允许 designer 覆盖 priority**（下拉改 High/Medium/Low），并记录 revision |
| 覆盖项入口 | — | ❌ 未做 | **新增**：「Coverage Items」可编辑列表，作为 Step 3 生成依据 |

**接口**：`POST /risk`（B）  
**演示要点**：指出高风险需求 → 手动将某条从 Medium 调为 High → 说明「designer 参与优先级决策」。

---

### Step 3 — Generate & Evaluate（FR 3.0 强制 + FR 4/5 加分）【核心】

**目标**：生成黑盒/白盒用例，支持审查、修订、追溯。

#### 3.1 黑盒用例区（FR 3.0，强制）

| 列 | 说明 | 可编辑 |
|----|------|--------|
| test_id | TC-AUT-xxx | 否 |
| requirement_id | 追溯需求 | 否 |
| technique | EP / BVA / DT | 否（或允许改） |
| title / test_steps / expected_result | 用例内容 | **是（强制交互）** |
| risk_level | High/Medium/Low | **是** |
| standard_ref | ISO 29119-4 引用 | 否 |
| status | Draft / Approved / Rejected | **是** |
| oracle / confidence | FR 5.0 | 只读 + 复核按钮 |

**必做交互（作业 Mainly 段落）**：

- 表格支持 **行编辑** 或 **侧栏编辑表单**
- 「批准 / 驳回 / 待复核」操作按钮
- 筛选：按 technique、risk_level、review 状态
- 显示 **traceability**：点击 requirement_id 高亮关联需求（Step 1 数据）

#### 3.2 白盒 FSM 区（FR 4.0，加分）

| 组件 | 说明 |
|------|------|
| Mermaid 渲染 | 展示 `stateDiagram-v2`（当前为 pre 文本，建议接 mermaid 库） |
| 覆盖路径列表 | all_states / all_transitions |
| 可选编辑 | 允许 designer 注释某条路径「已覆盖 / 未覆盖」 |

**接口**：`POST /fsm`（A）

#### 3.3 Oracle 区（FR 5.0，加分）

| 组件 | 说明 |
|------|------|
| llm_verdict / rule_verdict | 双判定展示 |
| confidence | 分档颜色标签 |
| needs_review | 触发「处理」按钮， designer 确认后改为 Approved |

**接口**：`POST /oracle`（B）

**演示要点（Step 3 整步）**：自动生成 3 条 EP/BVA/DT 用例 → Oracle 标出一条低置信度 → designer 修改 expected_result 并批准 → 展示 traceability 到 REQ-AUT-008。

---

### Step 4 — Optimize & Export（FR 7.0 加分 + FR 6.0 强制）

**目标**：优化套件并导出标准格式产物。

| 区域 | 组件 | 必做 | 建议增强 |
|------|------|------|----------|
| 优化模式 | Segmented：风险优先 / 标准 | ✅ 已有 | 切换时重新请求 `POST /optimize` |
| 对比 | Before / After 数量 + 缩减率 | ✅ 已有 | 列出被剔除的 test_id 列表（若 E 提供） |
| 导出 | JSON / CSV / XLSX 按钮 | ✅ 已有 | 导出前预览字段清单；仅导出 **Approved** 用例 |
| 修订摘要 | — | ❌ 未做 | 导出包内附带 revision log（前端生成 JSON 字段） |

**接口**：`POST /optimize`（E）、`GET /export/{format}`（A）

**演示要点**：优化前后对比 → 导出 CSV → 说明字段满足 Test Management Tool 导入需求。

## 四、全局状态与数据流设计

### 4.1 建议扩展 Zustand Store

当前仅有 `currentStep`，建议增加：

```typescript
// 建议结构（供实现参考）
{
  currentStep: number
  requirements: DisplayRequirement[]      // Step 1，含人工修订
  riskEntries: RiskEntry[]                 // Step 2，含人工覆盖的 priority
  coverageItems: CoverageItem[]            // 新增：覆盖项
  testCases: TestCase[]                    // Step 3，含 status / 修订内容
  oracleResults: OracleResult[]
  fsm: FSMResult | null
  optimizeResult: OptimizeResult | null
  revisions: RevisionLog[]                 // 全局修订记录（演示 Mainly 流程）
}
```

### 4.2 数据流原则

```
后端 AI 生成 → 写入 store（原始）
     ↓
Designer 在 UI 修订 → 写入 store（revised）+ revisions[]
     ↓
后续 Step 只读 revised 数据
     ↓
导出时带 revision 元数据
```

这样可以在演示中明确区分 **「AI 初稿」** 与 **「人工确认稿」**，直接对应作业评分中的 effectiveness / in-depth analysis。

## 六、演示视频（15 分钟）前端脚本建议

按作业 Presentation 要求，C 负责的 UI 演示建议占 **4-5 分钟**，顺序如下：

1. **开场（30s）**：展示 4 步 Stepper，说明测试的是 Library Management System（目标应用），不是工具本身。
2. **Step 1（1min）**：上传/粘贴借书需求 → 解析 → **手动修正一条 condition**。
3. **Step 2（45s）**：热力图指出高风险 → **手动调整一条 priority**。
4. **Step 3（2min）**：展示 EP/BVA/DT 用例 → FSM 路径 → Oracle 低置信度 → **编辑 expected_result 并批准** → 说明 traceability。
5. **Step 4（45s）**：优化前后对比 → 导出 CSV → 打开文件展示字段。
6. **收尾（30s）**：强调 interactive review 与 ISO 29119-4 技术覆盖。

## 八、设计决策摘要

| 决策 | 选择 | 理由 |
|------|------|------|
| 导航形态 | 左侧 4 步 Stepper | 符合作业流程叙事，演示清晰 |
| 是否拆 Dashboard/Workspace 多模块 | 否，保持单 Pipeline | 减少评委认知负担 |

| 加分 FR 位置 | 嵌入 Step 3/4 | 不增加顶级菜单复杂度 |
| 数据策略 | API 真调 + mock fallback | 联调前可演示，联调后自动 Live |
